# AuditCanvas 执行护栏

## 当前唯一北极星

在增加任何新功能或发布包装前，AuditCanvas 必须持续通过以下真实用户链路：

```text
扫描真实 Git 仓库
→ CLI serve 打开该运行的 Web Review Canvas
→ 展示每一处完整证据、路径和行号
→ 接受、拒绝或批注 Finding
→ 写入 .auditcanvas/reviews/
→ 刷新后恢复审阅状态
→ JSON/HTML 导出包含审阅结果
→ resolve-findings 读取 accepted Finding
```

## 完成证据

只有临时 Git 仓库的 Playwright 端到端测试同时证明以下事项，才能将画布闭环标记为已验证：

1. 页面显示“真实工作区”，而不是示例数据。
2. 三处重复内容全部完整显示，不含替代证据的省略号。
3. 审阅状态和人工批注写入版本化审阅文件。
4. 刷新页面后状态和批注保持不变。
5. 浏览器导出的运行包含合并后的审阅结果。
6. `resolve-findings` 能读取 Web 中接受的问题。

组件测试、静态截图、示例数据、README 描述、CI 通过或页面已部署，均不能单独作为产品完成证据。

## 禁止漂移

在北极星链路失败时，禁止开展以下工作：

- UI 美化、翻译扩写或营销页面。
- Release、Pages、Marketplace 或仓库元数据优化。
- 非阻塞依赖升级。
- AI Provider、PDF/DOCX、趋势看板或其他新功能。

允许的例外只有：直接解除北极星链路阻塞的安全修复和依赖修复。

## 状态标签

- **已验证**：真实端到端测试覆盖用户可见结果和持久化副作用。
- **部分实现**：存在组件、接口、原型或单独测试，但用户链路未闭合。
- **未实现**：没有可运行的用户入口。

不得使用代码数量、提交数量或发布状态替代产品状态。

## 改动准入

每项改动开始前必须同时回答“是”：

1. 是否直接推进或保护北极星链路？
2. 用户是否能在真实工作区观察到结果？
3. 不做这项改动是否会阻塞当前闭环？

否则不进入当前迭代。
